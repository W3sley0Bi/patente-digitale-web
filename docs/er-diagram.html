What's in this diagram
  • 3 existing entities (with renames/extensions): auth.users, profiles, driving_schools (renamed from claimed_schools; absorbs the old pending_claims via a status column).
  • 3 new tables: driving_licences, school_vehicles (FK'd to driving_licences.licence_id), school_photos.
  • Claim lifecycle: every school is one row whose status transitions pending → accepted | rejected. place_id uniqueness is a partial index — multiple pending claims allowed, only one accepted.
  • Role enforcement: SQL trigger gates driving_schools.user_id to autoscuola|admin.
  • Deferred: reviews, school_memberships, membership_invitations. Re-add together when there's an enrolled-student UI surface.
  • Cardinality: 1 owner → many schools (franchise/chain supported via user_id FK).
  • 1 view: school_completion — derived, never written to. Founding Partner gate = completion_pct >= 80 AND verified.
  • Filterable tag arrays on driving_schools (languages_spoken — enum-constrained, documents_required, instructor_specializations) → text[] + GIN. Items with per-row state (licenses, vehicles, services) → normalized tables.
  • Money stored as float8 (e.g. driving_licences.price). EUR-only — no currency column. UI renders via Intl.NumberFormat('it-IT', { style: 'currency', currency: 'EUR' }) to enforce 2-decimal rounding. Float math is not exact — apply rounding before sums or commissions.
  • Photos: bytes in Supabase Storage; only storage_path in DB. Never store bytes in Postgres bytea.

Pricing & vehicles — no longer JSONB

prices, vehicles, and time_to_license are now first-class rows in driving_licences and school_vehicles. Money is float8 on driving_licences.price — apply UI rounding before display.
